Contexte

Dans cette première mission nous allons d'abord prendre en main les données de KDou en étudiant la base de donnée et l'interrogeant sur excel.

Analyse globale de la base de données

Nous avons d'abord identifiés les clés primaires et secondaires :

  1. Clés primaires :
    1. Table Fournisseur : l’attribut « N° fournisseur »
    2. Table Produit : l’attribut « Réf produit »
    3. Table Categorie : l’attribut « Code catégorie »
    4. Table Clientele : l’attribut « Code client »
    5. Table Commande : l’attribut « N° commande »
    6. Table DetailCommande : la clé primaire par attributs d’ensemble ici est composée de l’attribut « Réf produit » et de l’attribut « N° commande ». Ces deux attributs permettent d’avoir une unicité.
  2. Clés secondaires
    1. Table Fournisseur : Aucun attribut
    2. Table Produit : les attributs « N° fournisseur » et « Code catégorie »
    3. Table Categorie : Aucun attribut
    4. Table Clientele : Aucun attribut
    5. Table Commande : l’attribut « Code client »
    6. Table DetailCommande : les attributs « Réf produit » et « N° commande »

On a donc le schéma de la BDR suivant :

FOURNISSEUR (N° fournisseur, Société, Contact, Fonction, Adresse, Ville, Région, Code postal, Pays, Téléphone)
PRODUIT (Réf produit, N° fournisseur, Code catégorie, Nom du produit, Quantité par unité, Prix unitaire, Unités en stock)
CATEGORIES (Code catégorie, Nom de catégorie, Description)
CLIENTELE (Code client, Société, Contact, Adresse, Ville, Code postal, Pays, Téléphone)
COMMANDE (N° commande, Code client, Date commande, A livrer avant, Date envoi, Port)
DETAILCOMMANDE (Réf produit, N° commande , Prix unitaire, Quantité, Remise (%))

Conclusion

Pour conclure sur cette première approche de la base de données de KDou, nous pouvons affirmer que les données ne sont pas de bonne qualité. En effet, pendant notre étude nous avons relevé une multitude d'erreurs et d'incohérences. Aussi, il était très compliqué de faire des recherches plus avancées sur la base de données étant donné que nous étions sur Excel et qu'il n'y a pas de contraintes d'intégrités.

Pour améliorer les données on peut mettre en place plusieurs petites choses concernant la structure même de la base de données, en voici quelques exemples :

  • Arrondir les remises pour que la donnée soit plus lisible.
  • Ecrire toutes les dates sous la forme jj/mm/aaaa pour faciliter les recherches dans la table.
  • Représenter chaque pays des clients et fournisseurs par une valeur pour faciliter les recherches dans les tables.
  • Remplir entièrement l'attribut "Région" dans la table FOURNISSEUR qui est très peu utilisées et pourrait servir pour des recherches plus précises et poussées.

Mais, il faudra surtout changer d'outils pour parcourir cette base de données en utilisant par exemple Microsoft Access, ce qui nous permettrait d'appliquer des contraintes d'intégrité et ainsi "lier" les tables entre elles et avoir un ensemble de données beaucoup plus cohérent. On pourrait appliquer les contraintes d'intégrités suivantes :

  • Contraintes d'intégrités (CI) de domaine :
    • « Prix unitaire » est un monétaire
    • « Date commande » et « A livrer avant » et « Date envoi » sont des dates(jj/mm/aaaa), « Port » est un monétaire
    • « Remise (%) » est un pourcentage
  • Contraintes d'intégrités (CI) déclarative :
    • « N° fournisseur » est une clé primaire et ne doit pas être NULL, « Société », « Contact », « Adresse », « Ville », « Code postal », « Pays » et « Téléphone » ne doivent pas être NULL
    • « Réf produit » est une clé primaire et ne doit pas être NULL, « Code catégorie », « Nom du produit », « Prix unitaire » et « Unité en stock » ne doivent pas être NULL
    • « Code catégorie » est une clé primaire et ne doit pas être NULL, « Nom de catégorie » et « Description » ne doivent pas être NULL
    • « Code client » est une clé primaire et ne doit pas être NULL, « Contact », « Adresse », « Ville », « Code postal », « Pays » et « Téléphone » ne doivent pas être NULL
    • « N° commande » est une clé primaire et ne doit pas être NULL, « Date commande », « A livrer avant », « Date envoi » et « Port » ne doivent pas être NULL
    • « Quantité » et « Remise (%) » ne doivent pas être NULL
  • Contraintes d'intégrités (CI) référentielle :
    • Les attributs « N° fournisseur » et « Code catégorie » dans la table PRODUIT doivent exister comme valeur de clé primaire « N° fournisseur » dans la table FOURNISSEUR et « Code catégorie » dans la table CATEGORIE
    • L’attribut « Code client » dans la table COMMANDE doit exister comme valeur de clé primaire « Code client » dans la table CLIENTELE
    • Les attributs « Réf produit » et « N° commande » dans la table DETAILCOMMANDE doivent exister comme valeur de clé primaire « N° commande » dans la table COMMANE et « Réf produit » dans la table PRODUIT
  • Contraintes d'intégrités (CI) d'entité :
    • Un FOURNISSEUR est identifié de manière unique par la valeur de « N° fournisseur », sa clé primaire
    • Un PRODUIT est identifié de manière unique par la valeur de « Réf produit », sa clé primaire
    • Une CATEGORIE est identifiée de manière unique par la valeur de « Code catégorie », sa clé primaire
    • Une CLIENTELE est identifiée de manière unique par la valeur de « Code client », sa clé primaire
    • Une COMMANDE est identifiée de manière unique par la valeur de « N° commande », sa clé primaire
    • Un DETAILCOMMANDE est identifié de manière unique par un couple composé de la « Réf produit » et de « N° commande » qui identifie la commande
